# Factry Historian Module - Implementation Steps

Incremental implementation plan with testing after each step to ensure everything works before moving forward.

## Implementation Order

### Step 1: Module Loads Successfully ✅
**Goal**: Module appears in Gateway → Config → System → Modules

**Implementation**:
- Verify the basic `GatewayHook` is working
- Module shows up as "Running" status
- No errors in gateway logs

**Test**:
- Install `.modl` file in Ignition Gateway
- Check Gateway → Config → System → Modules
- Verify status is "Running"
- Check `logs/wrapper.log` for any errors

**Success Criteria**: Module listed and running without errors

---

### Step 2: History Provider Appears in List 📋
**Goal**: See "Factry Historian" in Gateway → Config → Tags → History

**Implementation**:
- Implement a `HistoryProvider` class extending Ignition's base history provider class
- Register it in `GatewayHook.setup()` using `GatewayContext.getHistoryManager().registerProvider()`
- Provider shows in dropdown but doesn't need to work yet
- Just registration - no functionality needed

**Test**:
- Navigate to Gateway → Config → Tags → History
- Click "Create new Tag History Provider"
- See "Factry Historian" in the provider type dropdown

**Success Criteria**: Provider appears in list, can be selected (may show errors if configured, that's OK)

---

### Step 3: Can Query Data from Provider 🔍
**Goal**: Query historical data from provider endpoint

**Implementation**:
- Implement `HistoryProvider.readRaw()` method
- Call proxy's `/provider` endpoint via HTTP client
- Parse JSON response from proxy
- Convert proxy response to Ignition's `QualifiedValue` format
- Return data to Ignition

**Technical Details**:
- Use Java HttpClient to call http://localhost:8111/provider
- Map proxy's TagSample to Ignition's QualifiedValue
- Handle quality codes (192 = Good, etc.)
- Convert Unix milliseconds to Ignition timestamps

**Test**:
- Use script console: `system.tag.queryTagHistory(paths=["[default]TestTag"], startDate=..., endDate=...)`
- Check if random data comes back from proxy
- Verify quality codes are correct
- Check proxy console logs to see the query was received

**Success Criteria**: Can query and receive data, even if it's random generated data

---

### Step 4: Create Historian Instance and Configure Tags 🏷️
**Goal**: Configure tags to use Factry Historian

**Implementation**:
- Create a historian provider instance in Gateway UI
- Configure connection settings (proxy URL, etc.)
- Create test tags with history enabled
- Point tags to use "Factry Historian" provider
- Configure history settings (storage provider, deadbands, sample rates)

**Technical Details**:
- May need basic config UI or use tag property configuration directly
- Set historian provider on tag's history configuration
- For initial version, can hardcode proxy URL in code

**Test**:
- Create a tag in Tag Browser
- Go to tag's History configuration
- Enable history
- Select "Factry Historian" as provider
- Save tag configuration

**Success Criteria**: Tags successfully configured to use Factry Historian, no errors in logs

---

### Step 5: Send Tag Changes to Collector ✍️
**Goal**: Tag value changes get sent to collector endpoint

**Implementation**:
- Implement `StorageProvider` interface
- Subscribe to tag change events via TagManager
- Implement batching logic (collect changes over time window)
- Send batches to proxy's `/collector` endpoint
- Handle deadbands, sample modes, and quality codes
- Implement error handling and retry logic

**Technical Details**:
- Use TagManager.subscribe() for tag change notifications
- Batch samples (e.g., every 5 seconds or 100 samples)
- POST JSON to http://localhost:8111/collector
- Map Ignition QualifiedValue to proxy's TagSample format
- Respect tag's history configuration (deadbands, on-change vs polling)

**Test**:
- Write a value to a configured tag
- Check proxy console - should see "[COLLECTOR] Received X samples"
- Verify tag path, timestamp, value, and quality are correct
- Try changing multiple tags rapidly to test batching

**Success Criteria**: Tag changes appear in proxy console logs with correct data

---

## Why This Order?

**Original suggestion was**: 1 → 2 → 4 → 3 → 5
**Recommended order is**: 1 → 2 → 3 → 4 → 5

**Reasoning**:
- **Step 3 before Step 4**: Reading data is simpler than writing
  - No tag subscriptions needed
  - No event handling required
  - Can test immediately with script console
  - Don't need tags configured yet

- **Step 5 last**: Writing data is most complex
  - Requires tag change subscriptions
  - Needs batching logic
  - Deadband handling
  - More moving parts = more potential bugs

## Testing Tips

```python

system.tag.queryTagHistory(
    paths=["[default]TestTag"],
    startDate="2024-01-01 00:00:00",
    endDate="2024-01-01 01:00:00"

```

### Watch Gateway Logs
```bash
# Monitor logs in real-time
tail -f logs/wrapper.log
```

### Enable Debug Logging
Add to your GatewayHook:
```java
private static final Logger logger = LoggerFactory.getLogger(FactryHistorianGatewayHook.class);

@Override
public void setup(GatewayContext context) {
    logger.info("Factry Historian module setup starting...");
    // ... your code
    logger.info("Factry Historian module setup complete");
}
```

### Start with Hardcoded Config
For initial testing, hardcode the proxy URL:
```java
private static final String PROXY_URL = "http://localhost:8111";
```

No need for config UI in first iterations.

### Test Incrementally
- Don't move to next step until current step is solid
- Each step builds on the previous
- If something breaks, you know which step introduced the bug

## Additional Considerations

### Error Handling
- Gracefully handle proxy being offline
- Log all HTTP errors
- Return empty results rather than throwing exceptions

### Quality Codes
Map Ignition → Proxy:
- Ignition Good (192) → Proxy Good (192)
- Ignition Bad (0) → Proxy Bad (0)
- Ignition Uncertain (64) → Proxy Uncertain (64)

### Data Types
Proxy accepts:
- Numbers (int, float)
- Booleans
- Strings

Make sure to handle all Ignition tag data types.

### Performance
- Batch writes (don't send every tag change individually)
- Use connection pooling for HTTP requests
- Consider async/non-blocking HTTP calls
